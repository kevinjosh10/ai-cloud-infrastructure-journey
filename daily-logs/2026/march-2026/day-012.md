## 🔗 Pipes (`|`) — Connecting Commands

A **pipe** sends the output of one command directly as the input to another command.

**Syntax**

```bash
command1 | command2
```

This allows commands to work together like a **processing pipeline**.

**Example**

```bash
ls | grep ".txt"
```

Explanation:

- `ls` → lists files in the directory  
- `grep ".txt"` → filters only `.txt` files  

Another practical example used frequently in server environments:

```bash
ps aux | grep python
```

This command shows **running processes and filters those related to Python**.

---

## 🔗 Chaining Multiple Pipes

Linux allows multiple pipes to be chained together.

Example:

```bash
ls -la | grep '.txt' | wc -l
```

Step-by-step:

1. `ls -la` → lists all files with details  
2. `grep '.txt'` → filters `.txt` files  
3. `wc -l` → counts the number of lines  

Result: **Total number of `.txt` files in the directory**

## 📤 Redirection — Controlling Output

Redirection allows you to **send command output or errors into files instead of the terminal**.

### `>` Overwrite Output

Writes output to a file and **replaces existing content**.

```bash
ls > files.txt
```

This stores the directory listing inside `files.txt`.

---

### `>>` Append Output

Adds new content **to the end of a file** without deleting existing data.

```bash
echo "Day12 completed" >> progress.txt
```

Used commonly for **log files and progress tracking**.

---

### `<` Input Redirection

Reads input **from a file instead of the keyboard**.

Example:

```bash
wc -l < notes.txt
```

This counts the number of lines inside `notes.txt`.

---

### `2>` Redirect Errors

In Linux:

- `1` → standard output  
- `2` → error output  

Example:

```bash
ls fakefile 2> error.log
```

Instead of displaying the error on the terminal, it will be saved inside `error.log`.

This is very useful for **capturing system errors in scripts and server logs**.

## 🌍 Environment Variables

Environment variables store **important system values used by programs and scripts**.

They are accessed using the `$` symbol.

### Check PATH

```bash
echo $PATH
```

`PATH` tells Linux **where to search for executable commands**.

Example output:

```
/usr/local/bin:/usr/bin:/bin
```

---

### Check Home Directory

```bash
echo $HOME
```

Displays the current user’s **home directory**.

Example:

```
/home/user
```

---

### Create a Custom Environment Variable

```bash
export MY_VAR="hello"
```

Verify it:

```bash
echo $MY_VAR
```

Output:

```
hello
```

Environment variables are widely used for:

- API keys
- configuration values
- cloud credentials
- automation scripts

---

## 🐚 Shell Startup Files

When a terminal starts, Linux loads **shell configuration files**.

### `.bashrc`

Runs every time a **new terminal session** starts.

Common uses:

- aliases
- environment variables
- shell customization

Example:

```bash
alias ll="ls -la"
```

Now you can run:

```bash
ll
```

---

### `.bash_profile`

Runs **when a user logs into the system**.

Example cases:

- SSH login
- login shells

---

### `.profile`

A **fallback configuration file** used when `.bash_profile` is not present.

---

## ⚙️ Real Cloud Engineering Example

Analyzing server logs:

```bash
cat server.log | grep ERROR | wc -l
```

This command counts **how many errors exist in a server log file**.

Saving error output:

```bash
grep ERROR server.log > errors.txt
```

These techniques are heavily used in:

- **DevOps pipelines**
- **cloud server management**
- **log monitoring**
- **system debugging**

---

## 🧠 Key Takeaways

- Pipes (`|`) allow commands to work together.
- Redirection (`>`, `>>`, `<`, `2>`) controls where outputs and errors go.
- Environment variables store system configuration values.
- Shell startup files configure the terminal environment.

These skills form the **foundation of Linux automation**, which is essential for **cloud infrastructure engineering**.
