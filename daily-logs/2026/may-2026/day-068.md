## ✅ What I Completed

### 1️⃣ Understood the Pull Request Workflow

Learned the standard GitHub collaboration workflow:

    Branch/Fork
        ↓
    Make Changes
        ↓
    Commit Changes
        ↓
    Push Branch
        ↓
    Open Pull Request
        ↓
    Review & Feedback
        ↓
    Merge Changes

Understood:
- PR = request to merge code into another branch
- Teams review code before merging into `main`
- PRs improve collaboration and code quality

---

### 2️⃣ Learned Feature Branch Workflow

Created isolated feature branches instead of working directly on `main`.

Command used:

    git checkout -b feature/day68-pr-practice

Learned:
- Feature branches isolate changes safely
- Branch names should clearly describe purpose
- Professional naming conventions improve readability

Examples:

    feature/add-auth
    bugfix/login-error
    infra/terraform-vpc
    ci/github-actions

---

### 3️⃣ Practiced Git Commit & Push Workflow

Commands used:

    git add .
    git commit -m "docs: update README for PR practice"
    git push origin feature/day68-pr-practice

Learned:
- Proper commit messages are important
- Conventional commits improve project history
- GitHub automatically detects pushed branches for PR creation

---

### 4️⃣ Learned How to Write a Good PR Description

Understood professional PR structure.

Example format:

```markdown
## What
Updated README with setup improvements.

## Why
Makes onboarding easier for contributors.

## Testing
Verified markdown rendering locally.
```

Learned:
- PR descriptions explain changes clearly
- Helps reviewers understand purpose quickly
- Good documentation improves collaboration

---

### 5️⃣ Explored GitHub Pull Request Features

Learned important GitHub PR tools:

#### Reviewers
- People assigned to review changes
- Can approve or request modifications

#### Assignees
- Indicates who owns the PR

#### Labels
Examples:

    bug
    enhancement
    documentation
    infra

Used for organization and tracking.

#### Milestones
- Group issues/PRs into releases or sprints

#### Draft PRs
- Used for unfinished work
- Allows early feedback before final merge

#### Required Reviews
- Teams can enforce approvals before merging

---

### 6️⃣ Learned Code Review Etiquette

Understood how professional engineers review code respectfully.

#### Good Review Practices
- Be specific with feedback
- Suggest alternatives
- Explain reasoning
- Praise good code

Example:

    This could fail if the API returns null.
    Maybe add a null check here?

---

#### Learned `nit:` Suggestions

Example:

    nit: maybe rename this variable for readability

Meaning:
- Small optional improvement
- Not a blocking issue

---

### 7️⃣ Understood PR Review States

Learned reviewers can:

#### Approve
    Looks good to me

#### Request Changes
- Fixes required before merge

#### Comment
- Discussion without approval/rejection

---

### 8️⃣ Learned Merge Strategies

Explored different merge options.

#### Merge Commit
- Preserves full branch history

#### Squash and Merge
- Combines commits into one clean commit
- Keeps Git history cleaner

#### Rebase and Merge
- Maintains linear commit history

Learned:
- Squash merge is commonly preferred in teams

---

### 9️⃣ Learned Branch Cleanup After Merge

Commands used:

    git branch -d feature/day68-pr-practice

    git push origin --delete feature/day68-pr-practice

Learned:
- Merged branches should be deleted
- Keeps repositories clean and organized
