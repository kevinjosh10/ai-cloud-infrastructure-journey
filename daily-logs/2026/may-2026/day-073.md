## ✅ What I Completed

### 1️⃣ Learned What Security Groups Are

Understood that Security Groups (SGs) act as virtual firewalls for EC2 instances.

Learned:
- Security Groups control inbound and outbound traffic
- SGs are attached at the instance level
- Multiple EC2 instances can share the same SG

Visualized concept:
    User Request
         ↓
  Security Group Check
         ↓
     EC2 Instance

Understood:
- Security Groups are one of the most important AWS security features
- Every production EC2 environment relies heavily on SGs
- Proper SG configuration reduces attack surface significantly

---

### 2️⃣ Understood Stateful Firewall Behavior

Learned one of the most important Security Group concepts:
    Security Groups are STATEFUL

Meaning:
- Return traffic is automatically allowed
- Response traffic does not require explicit outbound rules

Example studied:
    Laptop → EC2 (SSH Request)
           ↓
    EC2 → Laptop (Response Allowed Automatically)

Compared behavior:
- Request allowed inbound
- Response automatically permitted outbound

Understood:
- AWS automatically tracks connection state
- Simplifies firewall management
- Reduces configuration complexity

---

### 3️⃣ Learned Security Group Rules Are Additive

Studied how Security Group rules combine together.

Learned:
- SGs only contain ALLOW rules
- There are NO DENY rules
- If traffic matches any allow rule → traffic is permitted

Example explored:

| Port | Source |
|---|---|
| 22 | My IP |
| 80 | 0.0.0.0/0 |

Result:
- SSH allowed from personal IP
- HTTP allowed from internet

Understood:
- Security Groups are permissive by matching rules
- Rule combinations determine final access behavior
- Careless configurations can accidentally expose services

### 9️⃣ Learned Security Group Best Practices

Studied secure SG management practices.

Best practices learned:
- Follow least privilege principle
- Open only required ports
- Restrict SSH access to trusted IPs
- Use SG references instead of wide CIDR ranges
- Separate SGs by server role

Compared:
    SSH → 0.0.0.0/0
vs
    SSH → My IP only

Understood:
- Overly permissive rules are dangerous
- Small mistakes can expose infrastructure publicly
- Good security design minimizes unnecessary access

### 🔟 Learned What EC2 Key Pairs Are

Studied EC2 authentication mechanisms.

Learned:
- Key pairs are used for secure EC2 login
- AWS stores the public key
- Users download and manage the private key

Authentication flow studied:
    Private Key
         ↓
 AWS Verifies Public Key
         ↓
    EC2 Access Granted

Understood:
- Key-based authentication is safer than passwords
- SSH keys are standard in Linux infrastructure

### 1️⃣1️⃣ Compared RSA vs ED25519 Keys

Studied different key algorithms supported by AWS.

---

#### RSA Keys

Learned:
- Older and widely supported algorithm
- Larger key sizes
- Commonly used historically

Examples:
- 2048-bit RSA
- 4096-bit RSA

---

#### ED25519 Keys

Learned:
- Modern elliptic curve algorithm
- Smaller and faster keys
- Strong security with better efficiency

Comparison studied:

| Feature | RSA | ED25519 |
|---|---|---|
| Age | Older | Modern |
| Key Size | Larger | Smaller |
| Speed | Slower | Faster |
| Security | Strong | Very Strong |

Understood:
- ED25519 is recommended for modern systems
- RSA remains widely compatible
- Key algorithm choice impacts performance and security

### 1️⃣2️⃣ Created Key Pairs Using AWS CLI

Practiced key pair creation commands.

RSA example:

      --key-name dev-rsa-key

ED25519 example:

      --key-name prod-ed25519-key \
      --key-type ed25519

Learned:
- AWS generates downloadable private keys
- Key type can be selected during creation
- Key pairs integrate directly with EC2 launch workflows

Understood:
- Key management is critical operationally
- Secure storage practices are essential

### 1️⃣3️⃣ Learned PEM File Security Practices

Studied secure handling of private keys.

Practiced:
    chmod 400 my-key.pem

Learned:
- PEM files should never be publicly shared
- AWS cannot recover lost private keys
- SSH may reject insecure file permissions

Understood:
- Losing PEM files may permanently lock access
- Private keys are equivalent to server credentials
- Strong operational discipline is required

---

### 1️⃣4️⃣ Studied Environment-Based Key Management

Learned how production environments separate keys.

Example strategy:
- dev-key
- staging-key
- prod-key

Understood benefits:
- Better access isolation
- Easier auditing
- Safer incident response
- Reduced blast radius

Learned:
- Shared keys are dangerous
- Separate environments require separate credentials
- Mature organizations rotate keys regularly
